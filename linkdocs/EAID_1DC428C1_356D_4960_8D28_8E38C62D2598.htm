For 2025, the cleanest ways to run a pre‑order form + take payment are:

Pick your path (from fastest to most flexible)

  1. 1. No‑/low‑code (fastest)
  • Stripe Payment Links: create a product (“Pre‑order: Widget v1”), share the link, or embed a button. Stripe handles checkout, 3‑D Secure (SCA), receipts, refunds.
    • Variants: collect full payment, a deposit, or save a card for later (via Setup Intent).
  • Typeform / Tally + Stripe: beautiful form + payment in one flow; perfect for collecting extra fields (size, color, notes) with a pay step at the end.
  • Shopify (with a pre‑order app) or Lemon Squeezy / Paddle: if you want storefront + VAT/tax handled for you (Paddle is a MoR, great for EU/UK VAT).
  1. 2. Hybrid (balanced power + speed)
  • Your React form + Stripe Checkout: host the form yourself, then redirect to Stripe Checkout (PCI‑safe, handles SCA). Store order data in MySQL before/after payment via webhooks.
  • Google Forms + Stripe Payment Link: ultra‑simple—collect details in the form, then auto‑redirect to a Stripe link. (Good for very early tests.)
  1. 3. Full‑code (maximum control)
  • Stripe Elements / Payment Intents inside your React app, Node backend, MySQL orders. Lets you do deposits, manual capture, save‑card-for-future precisely as you want.
    • For long pre‑orders: use Setup Intents (store customer’s payment method now, charge later on fulfillment). Manual capture on PaymentIntents only holds funds for ~7 days, so Setup Intents are the correct pattern for months‑long pre‑orders.


Recommended setup for you (practical + compliant)

React/Node/MySQL + Stripe Checkout + Webhooks

  1. 1. Product & pricing in Stripe (full price or deposit).
  2. 2. React pre‑order form (name, email, variant, quantity, notes) → POST to your Node API → write to preorders table.
  3. 3. Create Checkout Session server‑side; redirect user to Stripe.
  4. 4. Stripe Webhook (checkout.session.completed) → mark preorder paid, generate order number, send confirmation email.
  5. 5. Optional: use Setup Intent flow instead of taking money now (collect card + consent → charge when you ship).

Why: You get a polished, SCA‑compliant payment UI, strong fraud tools, and minimal PCI scope—while keeping all preorder metadata in MySQL for your admin dashboards.


Data model (minimal)

  • preorders(id, customer_email, name, variant, qty, notes, status, checkout_session_id, payment_status, created_at)
  • If using Setup Intents: store customer_id, payment_method_id securely (never raw card data).


VAT & compliance (UK/EU)

  • Decide when you recognize revenue (on charge vs on fulfillment).
  • If you collect now, you’ll issue the VAT invoice immediately. If charging later, Setup Intent + invoice on capture is cleaner.
  • If you don’t want to handle VAT complexities, use Paddle/Lemon Squeezy (Merchant of Record) or Shopify.


Nice extras

  • Inventory/limits: soft cap orders; waitlist beyond N units.
  • Email flow: confirmation + “we’ll notify you before shipping”; use HubSpot/MailerLite for updates.
  • Refund policy & ETA: show clearly on the form/checkout to reduce disputes.
  • A/B test price/deposit vs full pay using separate Stripe products/links.
